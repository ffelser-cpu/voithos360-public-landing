# Beta Milestone 2: Overview Page

## Goal

Create the authenticated patient overview page for the real beta app while keeping the same visual language as the prototype.

## Overview Page

- The overview page should be stylistically consistent with the prototype.
- The top left of the overview page must have a hamburger menu.
- The overview page must have a logout button.
- Clicking logout returns the user to the login screen.
- The overview page must have a "Start visit mode" button.
- Clicking "Start visit mode" takes the user to Visit Copilot so ambient recording can begin.
- The overview page must include top navigation tiles matching the prototype.
- Top navigation tiles must include Overview, Clinical, Open actions, Health gaps, Visit Copilot, Advocacy, and Caregivers.
- The Records tile should not appear in this beta navigation.
- The next tile on the overview page should display the user's name and a brief plain-English medical summary.
- The care complexity indicator should appear to the right of the user's name, matching the prototype.
- The summary should be generated by AI from EHR clinical data when available.
- Summary inputs should include the EHR problem list, past visit history, diagnoses, and other relevant clinical data.
- The summary must be 75 words or less.
- The summary should avoid medical abbreviations whenever possible.
- The patient-facing summary should address the user personally by name or with "you" and "your."
- The summary should briefly state what happened in the last visit and what Voithos360 is tracking.
- Just below the summary, the dashboard should display:
  - Open actions with a numeric count
  - Health gaps with a numeric count
  - Next visit as a countdown until the next scheduled provider visit, regardless of specialty
- Open actions and health gaps should be clickable and take the user to the corresponding section.

## Open Action Detection

- The system must use an open action detection and counting algorithm to identify unresolved tasks.
- Action sources include:
  - User tasks
  - Caregiver tasks
  - Care team tasks
  - Insurance tasks
  - Medication tasks
  - Appointment tasks
  - Administrative tasks
- The algorithm must exclude completed, cancelled, duplicate, expired, or resolved items.
- The system must display total open action count.
- The system must also count closed items.
- Open actions must be categorized by:
  - Type
  - Owner
  - Due date
  - Urgency
  - Status
- Action categories, owners, due dates, urgency tags, status tags, confidence scores, evidence scores, and rationale should be stored for reporting but not shown as visible tags in the patient-facing "What to do next" list.
- Inputs are derived from EHR data, clinical documents, tasks, messages, forms, claims, appointments, care plan items, and patient/provider communication recordings.
- LLM-assisted extraction should detect action items from clinical documents and Visit Copilot recordings.
- All action data, counts, exclusions, groupings, and patient actions must be stored in the backend database.

## Open Action Types

| Open action type | Example |
| --- | --- |
| Appointment | Schedule visit |
| Insurance | Follow up on prior authorization or other insurance-related issue |
| Claim | Appeal denied claim |
| Medication refill | Refill prescription |
| Records | Send records to specialist |
| Lab/imaging | Complete bloodwork |
| Care team message | Message provider |
| Cost | Find lower-cost option |
| Caregiver | Assign caregiver task |

## Status Rules

| Status | Counted as open |
| --- | --- |
| New | Yes |
| In progress | Yes |
| Waiting on user | Yes |
| Waiting on provider | Yes |
| Waiting on payer | Yes |
| Completed | No |
| Cancelled | No |
| Resolved | No |
| Duplicate | No |

## Open Action Algorithm Steps

1. Pull all tasks, messages, forms, claims, appointments, and care plan items.
2. Identify items with required follow-up.
3. Exclude completed, cancelled, duplicate, expired, or resolved items.
4. Group related items into a single action when appropriate.
5. Assign action type, owner, due date, severity, and status.
6. Count all remaining unresolved actions.
7. Display total open action count.
8. Display what needs to be done in the "Next best actions" section.
9. Color-code each action by severity, status, and urgency, matching the prototype.
10. Use plain-English language to describe what needs to be done.
11. Include a "Voithos360 found this..." line showing the record, plan, message, recording, or care activity that produced the action.
12. Display a "Get help" button and "Done" button beneath each action.
13. Clicking "Get help" should surface a summary of what Voithos360 prepared or did.
14. Where Voithos360 can take an action directly, it should prepare the message, call script, document, scheduling step, lab follow-up, insurance check, records request, cost check, or caregiver update for patient review.
15. If Voithos360 prepares a message or document, the user must see it before sending.
16. Prepared messages/documents must have an "Approve" button.
17. Clicking "Approve" stores the action history and marks the prepared item as sent.
18. If the action requires human support, payer/provider interaction, or information Voithos360 cannot complete directly, it should route the user to Advocacy.
19. Spreadsheet-based action rules should be used as the guideline for what can be automated, drafted, or escalated.
20. Clicking "Done" should gray out the item.
21. Store all items and every action taken in the backend database.

## Care Complexity Indicator

- The system must classify each user as:
  - Low Complexity
  - Moderate Complexity
  - High Complexity
  - Very High Complexity
- Each classification must include:
  - Numeric complexity score from 0 to 100
  - Top contributing drivers
  - Confidence score
  - Recommended support level
- Complexity score, confidence score, contributing drivers, scoring inputs, and domain scores must be stored but not displayed to the patient.
- The care complexity indicator must be color-coded:
  - Green: Low Complexity
  - Yellow: Moderate Complexity
  - Light red: High Complexity
  - Red: Very High Complexity

## Care Complexity Score Bands

| Text | Points | Meaning |
| --- | --- | --- |
| Low Complexity | 0-24 | Stable, needs few care tasks, low navigational burden |
| Moderate Complexity | 25-49 | Multiple appointments, some insurance/admin tasks, needs reminders |
| High Complexity | 50-74 | Multiple conditions, frequent care coordination, claims/prior authorization issues |
| Very High Complexity | 75-100 | Active care breakdown risk, caregiver burden, unresolved access/cost barriers |

## Complexity Inputs

- Number of active conditions
- Number of medications
- Number of specialists
- Recent emergency room visits or hospitalizations
- Prior authorization or claim denial activity
- Missed appointments
- Caregiver involvement
- Transportation or access barriers
- Cost concerns
- Patient confusion or repeated questions
- Open care gaps
- Number of unresolved tasks

## Stored Derived Data

- The AI-generated overview summary must be stored in the backend database.
- The complexity score, label, confidence score, top drivers, scoring inputs, domain scores, and recommended support level must be stored in the backend database.
- Stored derived data should support later dashboard display and analysis of how well Voithos360 manages individual patients over time.

## Measurable Impact / ROI Signals

- The overview page should keep the patient-facing "Measurable impact" section from the prototype.
- The system must store each measurable impact signal in the backend database.
- Impact signals should be useful both to the patient and later to a customer entity evaluating ROI.
- Each stored impact signal should include:
  - Category, such as avoidable delay, cost exposure, care blocker, medication safety, caregiver update, completed action, or admin burden reduced
  - Patient-facing label
  - Displayed value
  - Numeric value when applicable
  - Unit, such as count, dollars, days, tasks, or messages
  - Estimated dollar value when applicable
  - Calculation used to produce the displayed value
  - Source inputs used in the calculation
  - Linked care action IDs, health gap IDs, or other source records when available
  - Reporting period
  - Algorithm version
- Patient-facing ROI language should remain simple and concrete.
- Customer-facing reporting can later aggregate these stored impact signals across patients, time periods, categories, avoided delays, closed actions, clarified cost exposure, and completed advocacy work.
- The patient-facing app should keep measurable impact simple; detailed calculations remain stored for reporting.

## Lab Interpretation

- The system must read and interpret lab data.
- Lab interpretation must provide a plain-English summary to the user.
- The system should combine individual tests into one lab summary that creates a fuller picture of the user's overall health.
- Lab summaries should surface under "What happened recently" in the "Clinical" section.
- Lab summaries should state the summary and trend, not merely say a lab report was found or reviewed.
- Corresponding next steps should be added to "Next best actions."
- Each recommendation must have:
  - Confidence score
  - Evidence score
  - Rationale
- Confidence score, evidence score, and rationale must be stored in the database but not presented to the patient.
- Patient-facing language should be plain English and should explain what changed, why it may matter, and what to ask or do next.
- Appendix C, the lab interpretation engine, should guide detailed lab logic when available.

## Imaging Interpretation

- The system must read and interpret imaging report information.
- Imaging interpretation must provide a plain-English summary to the user.
- Imaging summaries should surface under "What happened recently" in the "Clinical" section.
- Imaging summaries should state the actual summary, not merely say an imaging report was found or reviewed.
- Any required follow-up should be added to "Next best actions."
- Imaging data should be pulled from radiology report text.
- Supported modalities include:
  - CT
  - MRI
  - X-ray
  - Ultrasound
- The system should extract:
  - Body region
  - Exam date
  - Indication
  - Impression section
  - Findings section
  - Prior comparison, if any
- The system should extract key findings:
  - Anatomy
  - Abnormality
  - Size
  - Laterality
  - Severity
  - Acute versus chronic
  - Incidental versus expected
  - Comparison to prior
- The system should classify findings into:
  - Normal/no acute findings
  - Acute actionable finding
  - Chronic/stable finding
  - Incidental finding
  - Follow-up recommended
- The system should determine trend:
  - New
  - Improved
  - Stable
  - Worsened
  - Resolved
  - No prior comparison
- The system should assign urgency:
  - Routine
  - Follow-up needed
  - Time-sensitive
  - Urgent/emergent
- The system should take action as suggested in Appendix D and summarize the report in plain English.
- In many cases, the impression section of the radiology report should provide enough information to create the patient-facing summary.
- Each imaging recommendation must have confidence score, evidence score, and rationale stored in the database but not shown to the patient.

## Hamburger Menu

- The hamburger menu must include two subcategories:
  - Settings
  - Preferences
- The menu should have an "X" close control in the top right.
- Clicking the close control returns the user to the overview page.

## Settings

- Settings should include an "Update your information" experience.
- The following fields should be pre-populated from intake and saved profile data:
  - First name
  - Last name
  - Date of birth
  - ZIP code
  - Email
- The user can optionally add/edit:
  - Caregiver name
  - Caregiver email
- Settings must include an optional provider list.
- Provider fields:
  - Provider name
  - Specialty
  - Organization
  - Phone
  - Source
  - Email
- Provider fields are free-text and editable.
- Provider data should be populated by AI from the EHR source when possible after EHR integration is connected.
- All settings data must be saved to the backend database.

## Insurance Cards

- Settings must show space for previously uploaded insurance card information.
- Settings must show the saved insurance card image when available, or provide a link to open it.
- The patient can upload a new insurance card.
- The patient can upload a secondary insurance card.
- The patient must be asked whether the uploaded card is primary or secondary.
- The self-pay checkbox belongs in the insurance card section, not beside caregiver fields.
- Insurance card information must be saved to the backend database.

## Caregiver Documents

- Any document or message intended for a caregiver must be shown to the user for approval before sending.
- Approved caregiver documents must be stored in the backend action history.
- Approved caregiver documents should also be visible from settings so the user can see what has been shared.

## Preferences

- Preferences should include reminder and escalation controls.
- Required controls:
  - SMS reminders yes/no toggle
  - Push notification reminders yes/no toggle
  - Urgent alerts only toggle
- Preference data must be saved to the backend database.
- Additional preference controls can be added in future milestones.

## Save Behavior

- A "Save settings" button appears at the bottom of the settings/preference flow.
- Clicking "Save settings" saves profile, caregiver, providers, insurance, and preferences to the backend database.
